## Spacing Scale

Existing tokens (multiples of 4) from Phase 1-2 spec, confirmed in codebase:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps inside buttons, badge dot offset |
| sm | 8px | Compact element spacing, inline padding, badge padding, step indicator gaps |
| md | 16px | Default element spacing, card padding (x/y), form field padding |
| lg | 24px | Section padding, progress panel inset, card gap from layout edges |
| xl | 32px | Layout gaps between major sections, ingest button offset from URL input |
| 2xl | 48px | Major section breaks, progress panel top/bottom padding |
| 3xl | 64px | Page-level spacing, full-page vertical centering |

Exceptions:
- **Sidebar width:** 256px (16rem, `w-64`) — existing standard app shell convention
- **Status badge height:** 20px (`h-5`) — compact badge sits inside card footer metadata row
- **Step indicator dot:** 8px (`size-2`) — inline with step label text
- **Progress panel min-height:** 120px — ensures visibility even when empty/loading
- **Repo input wrapper gap (6px `space-y-1.5`):** Compact vertical spacing inside repo URL input wrapper, tighter than `sm` (8px), keeps input, add button, and chip list visually close
- **Step row gap (10px `gap-2.5`):** Horizontal gap between step icon and step name — wider than `sm` (8px) for clear separation, more compact than `md` (16px)
- **Remove icon size (14px `size-3.5`):** Sits between `size-3` (12px, too small for click target) and `size-4` (16px, too large for inline chip removal)
- **Action row top margin (12px `mt-3`):** Vertical spacing between status row and action row in repo card — separates visually without the excess of `md` (16px)

### Usage Rules

- Primary (#4F46E5) reserved for **interactive elements only** — buttons, active nav, links
- Success/Warning/Risk colors used exclusively for **semantic indicators** — status badges, progress steps
- Info (#06B6D4) used for **in-progress states** only — queued/metadata fetching steps
- Status colors never used for decorative elements, borders, or non-semantic surfaces
- 60/30/10 rule: Background (60%), Surface/Cards (30%), Accent colors (10%)

## Interactions

### A. Repo URL Input + Add Repository (Create/Edit Forms)

```
Project Create/Edit Form Card
┌─────────────────────────────────────────────┐
│  Project Details                            │
│                                             │
│  Name [__________________________] *        │
│  Description [____________________]         │
│                                             │
│  Repository URLs                            │
│  ┌──────────────────────────────────────┐   │
│  │ https://github.com/owner/repo       │   │
│  └──────────────────────────────────────┘   │
│  ┌──────────────────────────────────────┐   │
│  │  Added Repositories:                 │   │
│  │  ● owner/repo [In Progress] [X]     │   │
│  │  ● owner/repo2 [Pending]    [X]     │   │
│  └──────────────────────────────────────┘   │
│  [+ Add Repository]                         │
│                                             │
│  Tags [...]                                 │
│                                             │
│  [Cancel]                     [Create]      │
└─────────────────────────────────────────────┘
```

1. **Initial state:** URL input empty, "Add Repository" button disabled, no repos listed. If project already has repos (edit mode), they render as chips/tags below input.
2. **User types URL:** Input validation runs on blur — must match `https://github.com/{owner}/{repo}` pattern. Invalid URL shows inline error below input.
3. **Click "Add Repository":** URL is validated, then added to a list below the input. Each list item shows `owner/repo` (parsed from URL), a status badge (always "Pending" initially), and a remove button (X icon). Input clears after successful add.
4. **Remove repo:** Click X on a repo chip. Repo is removed from the list. If ingestion has already started, show confirmation dialog: "This repository has an active or completed ingestion. Remove it anyway?"
5. **Maximum repos:** No hard limit enforced in UI (D-40 says no arbitrary limits). List scrolls if > 5 items.
6. **Error state (duplicate):** Inline error below input: "This repository has already been added."

```
Detail Page — Overview Tab
┌─────────────────────────────────────────────┐
│  Breadcrumbs: Projects › My Project         │
│                                             │
│  My Project                        [•••]   │
│  Created Jul 5, 2026  •  1 repo            │
│                                             │
│  [Overview] [Settings]                      │
│                                             │
│  ┌─ Repositories ─────────────────────────┐ │
│  │                                         │ │
│  │  owner/repo                             │ │
│  │  https://github.com/owner/repo          │ │
│  │                                         │ │
│  │  Status: ● In Progress                  │ │
│  │                                         │ │
│  │  [Ingesting…]                           │ │
│  │                                         │ │
│  │  ┌─ Ingestion Progress ──────────────┐  │ │
│  │  │  ✓ Queued                         │  │ │
│  │  │  ✓ Fetching Metadata              │  │ │
│  │  │  ● Cloning Repository       75%  │  │ │
│  │  │  ○ Analyzing Code                │  │ │
│  │  └──────────────────────────────────┘  │ │
│  └─────────────────────────────────────────┘ │
└─────────────────────────────────────────────┘
```

1. **Initial state (no repos):** "No repositories added yet." empty state with prompt to add URL. "Add Repository" form inline at top of section.
2. **Repo added, not ingested:** Shows repo card with URL, "Pending" status badge, enabled "Ingest Repository" button.
3. **Click "Ingest Repository":** Button transitions to "Ingesting…" (disabled, spinner), status changes to "Queued", backend triggers pipeline. Progress panel appears below.
4. **Ingestion running:** Button disabled (tooltip: "Ingestion already in progress"). Progress panel shows step-by-step status with active step animated. Polling fires every 3s.
5. **Ingestion complete:** Button changes to "Re-ingest". Status badge shows "Complete" (green). Progress panel shows all steps with checkmarks, final step shows completion timestamp.
6. **Ingestion failed:** Status badge shows "Failed" (red). Progress panel shows failed step with red X. Button stays "Retry" (D-50 checkpoint recovery). Error message below progress: specific failure reason.
7. **Rate-limit pause:** Status badge shows "Paused" (amber). Active step shows pause icon. Warning message: "GitHub API rate limit reached. Will resume in X seconds."

### C. Status Badge (Project Card)

```
Project Card
┌─────────────────────────────────┐
│  My Project                     │
│                                 │
│  Description of the project     │
│                                 │
│  📅 Jul 5, 2026  │  🌿 1 repo  │

```

1. The badge sits in the card footer, inline in the metadata row (replacing or alongside the "0 repos" text).
2. **No repos:** Badge not shown. Shows "0 repos" text unchanged.
3. **All repos complete:** Badge shows "Complete" (green background, white text).
4. **Any repo in progress:** Badge shows "In Progress" (cyan/indigo background, white text) — overrides other states.
5. **Any repo failed:** Badge shows "Failed" (red background, white text) — overrides "In Progress".
6. **Any repo paused:** Badge shows "Paused" (amber background, white text).
7. **All repos pending:** Badge shows "Pending" (muted gray background, secondary text).

**Priority for aggregation when multiple repos exist:** failed > paused > in_progress > complete > pending

### D. Inline Progress Panel (Project Detail Page)

Shown inside the repo card on the detail page when ingestion is active or has completed.

```
┌─ Ingestion Progress ────────────────────────┐
│                                             │
│  ✓ Queued                   2s              │
│  ✓ Fetching Metadata        4s              │
│  ● Cloning Repository       75%  ████████░░ │
│  ○ Analyzing Code                           │
│                                             │
│  Started: 10:32:45 AM                       │
│  Elapsed: 1m 23s                            │
└─────────────────────────────────────────────┘
```

1. **Layout:** Vertical step list. Each step has: status indicator (icon/dot), step name, optional metadata (duration, percentage), optional progress bar (for "cloning" step).
2. **Step indicators:** Completed = green checkmark (lucide `Check`), Active = indigo pulsing dot (lucide `Loader2` animated), Pending = gray circle outline (lucide `Circle`), Failed = red X (lucide `XCircle`), Paused = amber pause (lucide `PauseCircle`).
3. **Progress bar:** Only shown for "cloning" step when percentage is available. Thin (h-1.5), rounded, animated fill transition.
4. **Timestamps:** Each completed step shows duration on the right side. Top-level shows started time and elapsed time.
5. **Polling:** Poll `GET /api/projects/{id}/repos/{repo_id}/status` every 3 seconds while status is in a non-terminal state (pending/queued/fetching_metadata/cloning/analyzing). Stop polling on complete/failed/paused. On network error, keep polling with 5s backoff, show subtle "Unable to fetch status updates. Retrying…" below the panel.
6. **Empty state:** If no repos exist, section shows "No repositories added yet." with add-repo prompt.
7. **Skeleton loading:** During initial repo data load, show 3-4 skeleton step lines (pulsing muted bars) matching the progress panel layout.

```

│                                     │
│  This will create a new snapshot    │

│  existing evaluations. Continue?    │
│                                     │

1. **Trigger:** Click "Re-ingest" button on a completed repo.
2. **Primary action:** "Re-ingest" (primary indigo button), destructive variant not needed since old data is preserved.
3. **Cancel:** Closes dialog, no action.
4. **Loading state:** "Re-ingesting…" with spinner on the primary button after confirmation.

## New Components

### 1. `IngestionBadge` — `frontend/components/repos/ingestion-badge.tsx`

Props: `status: IngestionStatus` (union type). Renders a pill badge with the status color map above. Pure presentational, no polling logic.

### 2. `ReposField` — `frontend/components/repos/repos-field.tsx`

Props: `repos: Repo[], onChange: (repos: Repo[]) => void, disabled: boolean`. Reusable input component for add/remove repo URLs. Used in create form, edit form, and detail page settings tab. Includes URL validation (regex: `https?://github.com/{owner}/{repo}`), duplicate check, and inline error display.

### 3. `IngestionProgressPanel` — `frontend/components/repos/ingestion-progress-panel.tsx`

Props: `status: IngestionStatus | null, steps: StepStatus[], elapsed: string, error?: string`. Renders the step-by-step progress list with indicators, progress bar (for cloning), and timing footer. Accepts polling status data from parent. Pure presentational — no fetch logic.

### 4. `IngestButton` — `frontend/components/repos/ingest-button.tsx`

Props: `repoId: string, status: IngestionStatus, onIngest: (repoId: string) => Promise<void>, disabled?: boolean`. Renders the correct button state (Ingest / Ingesting… / Re-ingest / Retry) based on status. Handles click → confirmation dialog for re-ingest. Emits `onIngest` callback for parent to handle API call.

### 5. `useIngestionPolling` hook — `frontend/lib/use-ingestion-polling.ts`

Custom hook: `useIngestionPolling(repoId: string | null, enabled: boolean)` → `{ status: IngestionStatus | null, steps: StepStatus[], elapsed: string, error: string | null, isLoading: boolean }`. Implements REST polling at 3s interval. Stops on terminal status (complete/failed/paused). Handles backoff on network errors.

### 6. New API functions — `frontend/lib/api-repos.ts`

```

## Design Principles (Phase 3 Specific)

1. **Status at a glance** — Users should know ingestion state without opening detail page. Badge on project card serves this.
2. **Progress transparency** — Never show a spinner without a step label and context. Every active step shows what's happening.
3. **Error clarity** — Failed states always include: what failed, why (if known), and what to do next (Retry button).
4. **Non-blocking ingestion** — User adds URL, clicks Ingest, then can navigate away. Polling resumes on return. No blocking modals during ingestion.
5. **Checkpoint visibility** — On retry after pause/failure, show which step resumes from (D-50). Users see "Resuming from: Cloning Repository" instead of restart.
6. **Deterministic status** — Every status value maps to exactly one visual state. No ambiguity between "waiting" and "running."
